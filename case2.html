CASE 02 Live, built & deployed

ResuMatch: a three-LLM pipeline that tailors your resume to one job, not every job

Most resume tools rewrite tone, not substance. I built a working multi-agent pipeline- ATS keyword extraction, X-Y-Z bullet rewriting, and a tailored cover letter- chained across three coordinated LLM calls, to see what it actually takes to make a tool that reads the job description instead of ignoring it.

Why I built this

I wanted to actually build with agents and LLM chains, not just talk about them

This is my passion project- the one case study in this portfolio that isn't a strategy proposal but a real, working build. I wanted hands-on experience designing a multi-step LLM pipeline: how to split a task across specialized prompts, pass structured data between them, and handle the messy edge cases that any AI tool hits in practice (missing data, placeholder text, formatting drift).

The product question behind it is real too: most "AI resume builder" tools rewrite tone and formatting but don't actually engage with the specific job description in front of you. I wanted to test whether a tightly-scoped, ATS-aware pipeline could do better- extracting the actual keywords a given JD cares about, and writing toward them, rather than producing one generic "improved" resume regardless of role.

The pipeline

How the workflow is actually wired

Built and deployed in Dify. Three user inputs feed a document extractor, which passes parsed resume text to LLM 2 and LLM 3. LLM 1 runs in parallel on the job description only, and its keyword output feeds exclusively into LLM 2. All three LLM outputs converge at a single output block.

Deployed Dify workflow · solid lines = resume/text path · dashed lines = job description path · LLM 3 receives raw JD, not keyword list

The detail that took the most iteration: LLM 2 (resume rewrite) consumes both the Doc Extractor output and the keyword list from LLM 1, but LLM 3 (cover letter) bypasses LLM 1 entirely and reads the raw job description directly. That's deliberate, not an oversight: a resume benefits from dense, ATS-targeted keyword presence; a cover letter that's visibly keyword-stuffed reads as robotic to a human reader. Same job description, two different downstream needs, so the pipeline routes data differently for each.

Why a Doc Extractor node at all Resumes arrive as PDFs, Docx files, or pasted text- never clean structured data. The Doc Extractor node normalizes whatever comes in into plain text before any LLM sees it, so the downstream prompts can assume consistent input rather than each handling file-parsing edge cases themselves.
The prompts

System prompts, as actually deployed

These are the real prompts running in the live pipeline- not simplified for this write-up. I'm including them because the prompt design is the product here; the UI is a thin wrapper around three carefully scoped instructions.

LLM 1 · ATS Keyword Extractor
Gemini 2.5 Flash
You are an advanced Applicant Tracking System (ATS) parsing algorithm. Analyze the job description provided by the user. Extract exactly 10-12 technical hard skills and keywords mentioned. Format the output strictly as a comma-separated list. Do not include conversational filler. Job Description to analyze: {{Job_description}}
LLM 2 · Executive Resume Writer
Gemini 2.5 Flash
You are an elite Executive Resume Writer. Your task is to rewrite the user's current resume to align perfectly with their desired target role. Target Role: {{Target_Role}} Original Resume Data: {{Doc_Extractor.text}} ATS Keywords to integrate naturally: {{LLM.text}} CRITICAL EXECUTION RULES: 1. TRUTH & INTEGRITY: Never invent fake companies, modify real employment dates, or fabricate metrics. Reframe how existing experience is communicated. 2. THE X-Y-Z FORMULA: Rewrite professional experience bullet points using the formula: "Accomplished [X], as measured by [Y], by doing [Z]". Use strong, active action verbs. 3. KEYWORD INTEGRATION: Naturally integrate the provided ATS keywords seamlessly into the bullet points. Do not keyword stuff. 4. FORMAT: Output in clean, professional Markdown- standard headings (##), bold text (**), bullet points (*). No greeting or conversational fluff. 5. MISSING DATA HANDLING: If resume data contains placeholder text like "[Your Name]" or "[City, State]", do not output the bracketed placeholders. Omit the contact section entirely and start at ## Professional Summary. If graduation dates or university names are missing, write only the verified information (e.g. "Bachelor of Science" instead of "Bachelor of Science | [Your University]").
LLM 3 · Cover Letter Writer
Gemini 2.5 Flash
You are an expert career coach and professional copywriter. Your task is to write a highly compelling, tailored, 3-paragraph cover letter based on the user's resume and the job description. Original Resume Data: {{Doc_Extractor.text}} Target Job Description: {{Job_description}} STRUCTURE RULES: Paragraph 1: A powerful opening hook stating the applicant's enthusiasm for the role and why their background makes them a unique fit. Paragraph 2: Pick ONE impressive, metrics-driven achievement from the resume that directly addresses a primary requirement in the job description. Paragraph 3: A clear call to action expressing a desire for an interview, followed by a professional sign-off. EXECUTION RULES: 1. Do not invent any fake metrics or experience. Only use what is provided in the resume. 2. Output completely in clean Markdown format. 3. No introductory conversational fluff- start directly with the letter. 4. MISSING DATA HANDLING: If contact names, company locations, or personal data are missing, do not output bracketed placeholders like "[Company Name]" or "[Date]". Use professional generic styling or skip the header block, moving straight into "Dear Hiring Manager,"
The rule I'm most proud of Rule 1 in LLM 2 and Rule 1 in LLM 3 both explicitly forbid inventing metrics, companies, or experience. It would have been trivial to let the model "enhance" a thin resume with plausible-sounding fabrications- and early test runs without this constraint did exactly that. Writing the prompt to refuse fabrication, even when the honest output is less impressive, was a deliberate trade-off: a tailored resume that lies is worse than no tool at all.
The product

What the user actually sees

The Dify workflow sits behind a simple three-field interface built in Lovable: target role, current resume (paste or upload), and the target job description. One click runs the full pipeline and returns a tailored resume and cover letter side by side.

The output first appears as live-streaming text in the output interface, which the user can read and edit directly before doing anything with it. I built in manual editing functionality- the user can tweak wording, adjust bullets, or correct anything in the output panel itself before copying to clipboard or downloading the final document. This matters: the pipeline is a strong first draft, not a finished product, and giving users friction-free control over the last mile is what makes it actually usable.

Constraints

Building on a free-tier API budget

This whole pipeline runs on Gemini 2.5 Flash, used for all three LLM calls- a deliberate choice driven by token-budget reality, not by it being the ideal model for every step. Building under that constraint shaped real decisions:

Constraint → decision Using one fast, free-tier model for everything meant accepting a quality ceiling on the more nuanced writing tasks (the cover letter especially) in exchange for the pipeline being runnable end-to-end without a billing dependency. Each LLM node is configured to retry up to 3 times, with a 2-second interval between attempts. This is a pragmatic guard against two failure modes that come with free-tier API usage: transient rate-limit errors (where a short wait is enough to recover) and occasional model non-responses (where a fast retry is cheaper than surfacing an error to the user). Three attempts at 2s was the minimum that meaningfully reduced visible failures in testing without adding noticeable latency to a successful run.

The keyword-extraction step (LLM 1) is the cheapest, most deterministic task in the pipeline- strict comma-separated output, no creative writing- so it tolerates a smaller/faster model well. The resume and cover letter rewrites (LLM 2, LLM 3) are where model quality would matter most if this were scaled beyond a personal project.

What I'd improve

With better API access

If this moved beyond a personal/portfolio project and I had budget for paid API access, here's what I'd change first:

1

Split models by task

I'd keep Gemini 2.5 Flash (or something similarly fast and cheap, like GPT-4o mini) for the deterministic keyword-extraction step- it's a structured output task that doesn't need strong reasoning. For the resume rewrite (LLM 2) I'd move to Claude Sonnet or GPT-4o, which handle nuanced instruction-following and long-form writing quality significantly better. For the cover letter (LLM 3), where tone and persuasion matter most, I'd evaluate Claude Opus or Gemini 2.5 Pro- both show noticeably stronger voice and coherence on open-ended writing tasks in my testing.

2

Add a structured ATS-score output, not just keywords

Right now LLM 1 outputs a keyword list. With more headroom, I'd have it return a structured match score against the JD plus a gap list- so the user sees not just the tailored output but why it changed and what the original resume was missing.

3

Add a verification pass

A fourth LLM call (or a deterministic diff check) that compares the rewritten resume against the original to confirm no dates, employers, or metrics were altered- turning the "don't fabricate" prompt rule into something verifiable, not just instructed.

Reflections

What this taught me

The most useful lesson wasn't about prompt-writing technique- it was about data routing. Deciding that LLM 3 should see the raw job description instead of the keyword list LLM 1 produced was a small architectural choice with an outsized effect on output quality: it's the difference between a cover letter that reads like a person and one that reads like a keyword list wearing a trench coat.

The biggest open question I'd want real user data on: does a fully automated three-LLM pipeline actually produce a better tailored resume than a careful human editing pass with AI assistance, or does it just produce a faster one? I built this to find out hands-on what a multi-agent chain can and can't do well- and the honest answer right now is that it's very good at structure and keyword alignment, and still needs a human's judgment on what to emphasize.
